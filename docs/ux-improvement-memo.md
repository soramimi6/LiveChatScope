#### Q8 補足: フィルター変更時、どこまで分析をやり直すか

**背景（なぜこの質問が出たか）**

サマリー画面の「NG キーワード除外」「スタンプのみ発言除外」（UX-24）を ON/OFF すると、表示すべき **キーワード** と **話題ブロック** の中身が変わる。  
方針 Q7 では、フィルター変更のたびに **分析の再計算** を行う（画面表示だけの都度集計は使わない）。更新中は「更新中…」UI を出す。

分析処理は **複数の段階（ステージ）** に分かれている（`pipeline.py`）。主要なもの:

| 段階 | 処理内容 | 主に使う画面 |
|------|----------|--------------|
| 4 | キーワード抽出 | サマリー、話題分析 |
| 5 | 話題ブロック分割 | サマリー、話題分析、構成タイムライン |
| 6a | 話題の **切り替わり**（遷移）一覧 | 話題分析タブ |
| 6b | 話題ごとの **投稿者ランキング** | コミュニティタブ |
| 6c | 低活動区間 | 盛り上がりタブ（※チャット密度ベース → フィルター対象外） |
| 7 | サマリー画面用の **まとめデータ** | サマリータブ KPI・プレビュー |
| 8 | Markdown エクスポートファイルの生成 | エクスポート |

**論点**

段階 4・5 だけ再実行し、6・7 を **省略** すると:

- キーワードは新しいのに、サマリーの **話題プレビュー** が古いまま、など **画面間で内容が食い違う** 可能性がある。
- 話題分析の **遷移グラフ** が、新しい話題ブロックと一致しない可能性がある。

**選択肢**

| 案 | 内容 | メリット | デメリット |
|----|------|----------|------------|
| **狭い再実行** | 段階 4・5 のみ | 待ち時間が短い | 上記の食い違いリスク |
| **広い再実行（推奨）** | 段階 4・5 のあと **6a, 6b, 7, 8 も連続実行**（6c は密度ベースのため **省略**） | 全画面・エクスポートが **一貫** | 待ち時間はやや長い |

**ユーザーへの確認文（そのまま使える）**  
「フィルターを変えたあと、話題ブロックに連動する表示（サマリーまとめ、話題の切り替わり、話題別投稿者、Markdown エクスポート）も **まとめて再計算** してよいですか？ 推奨は **はい（広い再実行）** です。」

## 2. 実装との差分（コード照合 2026-06-21）

設計ドキュメントと現行コードが一致していない点。改修時の前提として参照。

| 項目 | 設計・期待 | 現行実装 | 影響 UX |
|------|-----------|----------|---------|
| `analysis_status=partial` | Phase A のみ完了（[architecture.md](architecture.md)） | Pipeline は常に `complete` で終了（`partial` をセットするコードなし） | `PartialAnalysisBadge` は実質未使用 |

| 動画メタ | タイトル・チャンネル・尺 | `fetch_worker.py` は **messages のみ** 保存。`title` / `channel_name` / `duration_seconds` は **UPDATE なし**（常に null になりうる） | ヘッダタイトルが `動画 {id}` フォールバック、UX-09 ブロック |
| `/summary` 件数上限 | `analysis_defaults.json` stage7: keywords **10**, topics preview **6**, highlights **5** | `analysis.py` が highlights / keywords / topic preview すべて **`LIMIT 5` 固定** | UX-07, UX-08 |
| JSON export `authors` | — | `author_stats` 全行出力だが、Stage 1 は **Top 20 のみ** 保存（`author_top_n=20`） | 「全投稿者」と誤解しやすい |
| markdown-summary | — | **API**（`export.py`）と **Stage 8 キャッシュ**（`stage8.py`）で内容不一致。DL は API 版 | キーワード・話題はキャッシュのみ |
| 収益タブ CSV | スーパーチャット専用想定（UI 文脈） | 同一 API `export/csv` → **全 messages** CSV | UX-23 と関連 |
| `tokens` テーブル | Stage 4 中間データ | `delete_tokens_after_stage4: true` で **Stage 4 後削除** | 再トークナイズは messages からのみ |
| `GET /videos/{id}` | — | `VideoMetaResponse` に **`source_url` なし**（DB カラムは存在） | UX-27 は API 拡張が必要 |

#### UX-23: JSON / CSV 役割分担 📋 ✅方針確定（Q2）

**方針（Q2）**: **JSON = 分析結果一式**、**CSV = messages 専用（チャットログ）**。ユーザー向け **説明 UI を追加**（要検討だが必須要件に近い）。

**確定後の役割**

| 形式 | 役割 | 含める主なデータ |
|------|------|-------------------|
| **JSON** | 分析結果の **アーカイブ / 連携用** | メタ、密度、Top 投稿者、SC、messages、（将来）highlights / topics / keywords 等。フィルター ON 時は messages・話題系に反映（UX-24） |
| **CSV** | **チャットログ**（表計算・検索用） | messages 列のみ（`time`, `author`, `type`, `text`, `jump_url` 等）。集約データは **含めない** |

**説明 UI 案（要検討）**

| 置き場所 | 内容例 |
|----------|--------|
| ExportMenu | `JSON — 分析結果一式` / `CSV — チャットログのみ`。各項目に 1 行 description |
| ツールチップ | 「CSV には密度・話題は含まれません。一式が必要な場合は JSON を利用してください」 |
| 収益タブ | 現状 CSV が全 messages になっているため、UX-23 実装時に **SC 専用 CSV** または文言修正とセットで整理 |

**現状との差分**

| 項目 | 現状 | 改修後 |
|------|------|--------|
| JSON | messages なし | messages + 不足している派生データを追加 |
| CSV | messages のみ（実質 OK） | 役割を明示。収益タブ経由も同じ定義に統一 |
| 収益タブ CSV | 全 messages API 共用 | SC 一覧 CSV か messages CSV かラベルで区別 |
